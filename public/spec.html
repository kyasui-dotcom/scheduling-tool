1. 概要とアーキテクチャ

Google Calendar と直接連携する社内向け予約調整ツール。Calendly ライクの体験を、複数名同席・イベント別営業時間・スプレッドシート自動出力など B2B 営業の実運用に必要な機能で拡張。

技術構成

フロントエンドNext.js 16 App Router / React 19 / Tailwind CSS 4 / shadcn/ui
認証NextAuth v5 (Google OAuth) + Drizzle Adapter
データベースNeon PostgreSQL (Serverless HTTP driver via drizzle-orm/neon-http)
外部 APIGoogle Calendar / Google Sheets / Slack Incoming Webhooks / Zoom
デプロイVercel (GitHub 連携で自動)

2. 認証・権限

ログイン

  • Google OAuth (NextAuth v5)
  • 要求スコープ: openid / userinfo.email / userinfo.profile / calendar.events / calendar.freebusy / spreadsheets
  • 初回サインイン時: events.createUser で username 自動生成 (email localpart) + デフォルト稼働時間 (平日 9:00-17:00) 作成
  • Session strategy: database (DrizzleAdapter)

権限モデル (同ドメイン共有)

組織テーブルは持たず、メールドメインで同組織を判定。lib/auth-helpers.ts:

  • getEmailDomain(email) — @以降を lowercase 抽出
  • canManageEventsOf({viewerUserId, targetUserId}) — 同じ非空ドメインなら true
  • getManagedUserIds(viewerUserId) — self + 同ドメイン全員の user_id 配列

代理管理

  • イベント作成時に「オーナー」を同ドメインユーザーから選択可
  • 選択したオーナーの Google カレンダーに予約イベントが作成される
  • 編集/削除も同ドメインなら可能
  • イベント一覧では「代理管理」ラベルで自他の区別を表示
制限: コンシューマドメイン (gmail.com など) を allowlist/blocklist で除外する仕組みは未実装。単一組織運用を前提。

3. イベントタイプ設定

3.1 基本情報

項目型説明
titlevarchar(255)公開表示名
slugvarchar(128) 乱数10文字URL のパス部分。新規作成時に crypto.getRandomValues で自動生成、再生成ボタンで rotate 可
descriptiontext公開説明文 (ゲストにも表示)
durationMinutesinteger所要時間
colorvarchar(7)UI ラベル色
isActivebooleanfalse でリンク404
meetingPlatformenumgoogle_meet / zoom / none

3.2 スケジューリング方式

specific_person指定メンバー1人 (オーナー) の空き時間で予約
any_available複数メンバーのいずれか空いてる人 → 予約時にランダムアサイン (公平抽選)
all_available全メンバー空いてる時間のみ予約可

any_available のランダムロジック: 予約数最少のメンバー同数の中から Math.random で抽選 (バイアスなし・round-robin公平性維持)

3.3 スロットモード

fixed_slotsdurationMinutes 刻みの固定スロットから選択
flexible_start空き時間帯の中から 15分刻みで開始時刻を自由選択

3.4 予約可能期間

  • 相対: minNoticeMinutes (最低通知時間) / maxAdvanceDays (先送り上限)
  • 絶対: bookingWindowStart / bookingWindowEnd (YYYY-MM-DD) — 設定時は相対より優先
  • キックオフ等「明日から6/26まで」の期間限定URL用

3.5 バッファ

  • bufferBeforeMinutes / bufferAfterMinutes — 予約前後の余白
  • 移動時間や休憩を挟む用途

3.6 カスタム質問

ゲスト予約フォームに追加する質問を JSONB で保持。event_type.custom_questions:

  • タイプ: text / textarea / radio / checkbox / select / phone / number
  • 必須フラグ / description / placeholder / options (radio 系のみ)
  • CustomQuestionEditor コンポーネントで編集

3.7 カレンダーイベントタイトル形式

calendarTitleFormat (text, default {title}{company}/{name}様) — 自由入力テンプレート

  • プレースホルダー: {title} (イベント名) / {company} (社名) / {name} (担当者名)
  • 編集画面にリアルタイムプレビュー
  • 旧プリセット名 title_first / company_first / company_only も後方互換で解釈

4. 営業日時と不在期間

4.1 ユーザーの週間スケジュール

/availability ページで設定 (availability_schedule + availability_rules)

  • タイムゾーン (デフォルト Asia/Tokyo)
  • 曜日ごとに 開始時刻-終了時刻 を複数設定可
  • 全イベントのデフォルト稼働時間として使用

4.2 イベント別営業日時

event_type.business_hours (JSONB) — 設定時はユーザー週間ルールを上書き

[{ dayOfWeek: "monday", startTime: "10:00", endTime: "17:00" }, ...]
  • null → ユーザーの週間スケジュール使用 (既存挙動)
  • [] → 全曜日で予約不可
  • [items] → イベント専用に上書き

4.3 不在期間

availability_override テーブル — 特定日を予約不可に

  • 日付範囲を /availability ページから登録
  • 1日1行で保存、UI では連続日を range にグルーピング表示
  • 該当日の予約フォームでは日付ボタンがグレーアウト

4.4 空き時間計算の優先順位

1. availability_override (isBlocked=true) 最優先。日単位で完全ブロック 2. event_type.business_hours 設定時はユーザー週間を上書き 3. availability_rules (曜日ルール) 上記が両方無ければコレ 4. Google Calendar FreeBusy 窓から busy を差し引き 5. 既存の confirmed / held booking + buffer 更に差し引き

5. 予約フロー

5.1 ゲスト側の体験

1. 公開URL /b/<slug> を開く - Server Component が今日+明日の空きを prefetch → 即描画 2. カレンダーで日付選択 → その日の空きスロットを表示 3. スロットをクリック - client 側: cache から availableUserIds → ランダムで assignee 決定 - POST /api/bookings/hold で 10分間の仮押さえ - URL: /confirm?slot=...&assignee=...&hold=<uuid> 4. 確認画面: 社名 / 担当者名 / メール (すべて必須) + カスタム質問 5. POST /api/bookings で確定 - 仮押さえを confirmed に UPDATE (INSERT せず) - createCalendarEvent → Google Calendar に予定作成 - Sheets 追記 / Slack 通知 (設定時) 6. /booking-success/[bookingId] にリダイレクト - .ics DL / Google Cal 追加リンク / 予約管理URL 表示

5.2 予約時に必須入力

  • 社名 (guestCompanyName)
  • 担当者名 (guestName)
  • メール (guestEmail)
  • + カスタム質問の required=true のもの

5.3 Google Calendar イベント内容

  • タイトル: calendarTitleFormat テンプレートを置換
  • Description: 顧客情報 / 会議URL / 顧客メモ / カスタム質問回答が全て入る
  • Attendees: [data.guestEmail] のみ、responseStatus: "accepted" でゲストは自動参加扱い
  • Reminders: 24h前 email + 30min前 popup (Google 側)
  • Google Meet の場合は conferenceData で Meet URL 自動発行

5.4 リスケジュール

  • ゲストが /booking-manage/[bookingId]?token=<guestToken> から新日時選択
  • 元予約をキャンセルして新予約作成 (URLに ?reschedule= + token を付与して confirm へ)
  • guest_token は lib/guest-token.ts の HMAC で生成 (改ざん防止)

5.5 キャンセル

  • ゲスト: 同じ管理URLからキャンセル理由付きで実行
  • 担当者: /bookings 管理画面 → 個別予約から
  • Google Calendar のイベントも同時削除

6. 仮押さえ (Hold)

日時選択の瞬間に short-lived DB ロックを取得し、他ユーザーの二重予約を防止。

スキーマ

  • bookings.status enum に held 追加
  • bookings.held_until (timestamptz) — ロック期限
  • デフォルト HOLD_MINUTES = 10 (定数 in /api/bookings/hold/route.ts)

availability への反映

空き時間計算で busy と扱う条件:

status = 'confirmed'
  OR (status = 'held' AND held_until > NOW())

失効した held は自動的にクエリから除外され、実質解放される。cron 不要。

フロー

日時クリック → POST /api/bookings/hold { eventTypeId, startTime, assignedUserId, guestTimezone } → 該当スロットで overlap する confirmed / active-held を確認 → 無ければ status='held' の row を INSERT (guestは仮の値) → { holdId, heldUntil, holdMinutes: 10 } 同時アクセス [ゲストB] → availability クエリでは held が busy 扱い → 同枠選択不可 10分以内に確定 → POST /api/bookings (holdId 付き) → 既存 held 行を UPDATE で confirmed + 実 guest 情報に格上げ 10分経過 (離脱) → held_until < NOW() → クエリで除外 → 実質解放

8. 予約管理画面 (/bookings)

表示

  • 同ドメインメンバーの予約を横断で一覧 (getManagedUserIds でスコープ)
  • 50件/ページ (PAGE_SIZE = 50)
  • 各予約行の下にカスタム質問の回答を Q: A 形式でサブ行表示

列

予約日時 / 開始日時 (開始-終了) / 担当 / 顧客 (社名 + 担当者名 + メール) / イベント / 状態 / 会議リンク

フィルタ

  • スコープ: 自分のみ / 同ドメイン全員
  • 担当者: 全員 / 個別
  • ステータス: confirmed / cancelled / all
  • 開始日 from-to (絶対日付)

CSV エクスポート

右上「CSV エクスポート」→ 現在のフィルタ条件を維持して /api/bookings/export

  • UTF-8 BOM + CRLF (Excel で日本語文字化けなし)
  • JST 表記
  • 200件上限なし (該当全件)
  • 18列: 予約作成日時 / 開始 / 終了 / 所要分 / 担当者名 / 担当者メール / 顧客社名 / 顧客担当者名 / 顧客メール / イベント名 / 会議PF / 会議URL / ステータス / キャンセル日時 / キャンセル理由 / 顧客メモ / TZ / カスタム質問回答 / 何回目

10. スプレッドシート排出 (/exports)

フィルタ条件付きの排出タスクを複数管理。手動実行で該当予約を Google Sheets に一括追記。

タスクの設定項目

nameタスク名 (例: 週次商談レポート)
spreadsheetUrlGoogle Sheets の URL (#gid= でタブ固定可)
sheetNameシート名 (省略時は1枚目)
eventTypeId特定イベント or 全イベント
assigneeUserId特定担当者 or 全員
statusconfirmed / cancelled / all
daysBack直近N日 (絶対日付より優先)
fromDate / toDate絶対日付範囲
includeHeader実行のたびにヘッダ行を先頭に追加
lastRunAt / lastRunStatus / lastRunRowCount / lastRunError最終実行結果 (自動更新)

出力列 (15列)

予約日時 / 開始日時 / 終了日時 / 所要分 / 担当者名 / 担当者メール / 顧客社名 / 顧客担当者名 / 顧客メール / イベント名 / 会議URL / 顧客メモ / カスタム質問回答 / ステータス / 何回目

実行フロー

POST /api/exports/[taskId]/run → タスクのフィルタで bookings を SELECT → buildBookingRow() で各予約を15列配列に変換 → includeHeader=on なら BOOKING_ROW_HEADER を先頭に追加 → appendRowsToSheet() で 1 API 呼び出しで一括追記 → last_run_* を UPDATE

イベント画面との連携

  • イベント編集画面に「このイベントの排出タスク」カード
  • そのイベント (eventTypeId 一致) のタスクを一覧表示、実行/編集/削除可
  • 「＋ 新規作成」→ /exports?new=1&eventTypeId=<id> で eventTypeId プリフィル

11. カレンダー / メール / Slack

11.1 Google Calendar への書き込み

  • 予約確定時に assignedUserId のカレンダーに events.insert
  • Meet URL を自動発行 (meetingPlatform = google_meet)
  • ゲストは attendees + responseStatus=accepted で自動参加
  • 他のメンバーの Google Calendar は触れない (attendee なし)

11.2 スプレッドシート自動追記

  • event_type.spreadsheetUrl が設定されていれば、予約確定と同時に1行追記
  • 認証: GOOGLE_SPREADSHEET env のサービスアカウント (JSON or base64 JSON) を優先、無ければ SHEETS_WRITER_EMAIL ユーザーの OAuth
  • URL の #gid=X でタブ固定 (省略時は1枚目)
  • 空タブに初回書き込み時はヘッダ16列を自動挿入
  • 16列: 予約日時 / 開始日時 / 終了日時 / イベント名 / 社名 / 担当者名 / メール / 顧客メモ / カスタム質問回答 / 会議URL / カレンダー担当者 / カレンダー担当者メール / 所要分 / 会議PF / 予約ID / 予約管理URL
  • 失敗しても予約は成立 (try/catch non-blocking)

11.3 Slack 通知

予約確定時に webhook で通知。優先順位 3段:

  1. event_type.slack_webhook_url (イベント個別)
  2. process.env.WEBHOOKURL (全体フォールバック)
  3. 両方未設定 → 通知なし

11.4 リマインダー

  • Google Calendar 側で 24h前 email + 30min前 popup
  • 独自リマインダーメールは未実装 (将来拡張余地)

11.5 予約ID / 予約管理URL

  • 予約ID = booking.id (UUID)
  • 予約管理URL = <appUrl>/booking-manage/<id>?token=<guestToken>
  • guestToken は HMAC (改ざん防止) - lib/guest-token.ts

12. ユーザーデフォルト

/settings の「イベント作成時のデフォルト」カードで設定。新規イベント作成時にフォームへプリセットされる。

対象項目 (users テーブル)

  • defaultSlackWebhookUrl
  • defaultSpreadsheetUrl
  • defaultCalendarTitleFormat
Slack Webhook の実効フォールバックは イベント個別 → ユーザーデフォルト → 環境変数 WEBHOOKURL の3段になっており、Slack 通知漏れを防ぎやすい。

13. 埋め込みウィジェット

public/widget.js

  • 1つの script タグで動作するタグ設置型ライブラリ
  • data-schedule="URL" 属性のボタン → クリックでモーダル (iframe)
  • data-schedule-inline="URL" 属性の div → その中にカレンダー iframe を挿入
  • MutationObserver で SPA の動的追加にも自動対応
  • ESC / 背景クリック / × ボタンでモーダル閉じる
  • window.Scheduler.popup(url) / Scheduler.inline(el, url) の JS API

EmbedDialog で提供される4つのコード

  1. モーダル予約タグ — script + button (ページ遷移なし、ポップアップで予約)
  2. カレンダー表示タグ — script + div (指定要素にインライン表示)
  3. 予約ボタンリンク — a タグ (別タブで予約ページ遷移)
  4. 直接リンク — URL コピー

イベント一覧・イベント編集画面のどちらからでも取得可能。

17. パフォーマンス最適化

17.1 サーバーサイド Prefetch

  • 公開予約ページ (/b/[slug] 等) の Server Component が今日+明日の空きを事前計算
  • BookingClient は initialAvailability prop から cache を seed
  • マウント時の初回 fetch を省略 → 初回描画が Google FreeBusy 待ちなく即時

17.2 スロット選択時 pre-assign + 仮押さえ

  • クライアント: cache から availableUserIds → ランダムで assignee 決定
  • /api/bookings/hold で 10分間ロック
  • POST /api/bookings は holdId + assignedUserId を受け取り、Google FreeBusy 再検証をスキップ
  • 予約確定時間を体感で 1秒以上短縮

17.3 空き時間計算の高速化

  • getAvailabilityRangeFromConfig: N日をまとめて 1回で計算
  • FreeBusy: 全ユーザー並列 (Promise.all), 期間は 1 API 呼び出しでカバー
  • schedules / rules / overrides / bookings は inArray でバッチ取得
  • 7日分プレビューでも Google API 呼び出しは N ユーザー分のみ (以前の 7N から削減)

17.4 プレビュー API での並列プリフェッチ (ブッキング画面)

  • Phase1: 今日+翌日を初期表示のために取得
  • Phase2: 表示中の月全体をバックグラウンド取得
  • ホバー時: 未キャッシュ日を先取り

17.5 Sheets 追記の非同期性

  • 予約 INSERT 後に Sheets append 失敗しても予約は成立 (try/catch)
  • スループット重視 (完全性は排出タスクの手動実行でカバー)
